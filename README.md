# PD Copywriter

> 每个词的选择都是产品决策——"才"和"就"可以翻转用户对同一件事的感知。

中文互联网产品文案引擎——基于语言学规则和用户心理学，帮助 PM 和设计师写出更好的中文产品文案。

## 为什么需要这个 Skill？

中文产品文案的坑比你想的深：

- **"才"和"就"是框架工程**——"3步才能完成"让用户觉得门槛高，"3步就能完成"让用户觉得简单。同一个事实，一个虚词翻转感知
- **"可"和"可以"冷暖差两档**——"你可领取"在利益场景偏冷硬（WARMTH=-2），"你可以领取"正常温暖
- **"确定/取消"是偷懒**——用户不看标题就不知道点了会发生什么。按钮应该写"撤回/不撤回"
- **"高达99.4%"对失败用户不真诚**——刚失败过的用户看到"高达"会觉得你在炫耀，而不是在帮他

这些不是"语感"问题，是有语言学和心理学依据的产品决策。PD Copywriter 把这些规则参数化，让文案选择从"凭感觉"变成"有依据"。

## 三类文案，规则不同

| 类型 | 代号 | 核心特征 | 典型载体 |
|------|------|---------|---------|
| **UI 交互文案** | UI | 短、即时、操作导向 | 按钮、Toast、弹窗、空状态、表单 |
| **营销推送文案** | MKT | 利益驱动、转化导向 | Push、短信、邮件主题、Banner |
| **说明型文案** | EXP | 完整、精确、合规导向 | 法律声明、活动规则、帮助中心 |

三类文案在标点、敬语、叠词、语气强度上规则完全不同——UI 禁用感叹号和"请"，MKT 利益场景可用，EXP 必须用句号且禁用感叹号。

## 核心能力

### 7 个语言学子模块

| 模块 | 解决什么问题 |
|------|------------|
| 语法模板库 | 正向/负向框架选择、"才"vs"就"、主语省略规则 |
| 虚词选择引擎 | "可"vs"可以"、"需要"vs"必须"、"请"vs 省略 |
| 动词选择矩阵 | "删除"vs"移除"vs"清理"——力度匹配不可逆性 |
| 主语选择规则 | 系统状态省略主语、成功用"为你"、合规用"我们" |
| 叠词开关 | 叠词 = FORCE-2, WARMTH+2，空状态可用，金融禁用 |
| 英→中适配 | Must→"需要"、Can→按焦点区分、Just→"只需/只要" |
| 韵律检查器 | 冗余词、空洞形容词、双重否定、字数对称 |

### 13 个场景端到端模板

错误提示、操作确认、空状态、功能引导、CTA/卖点、权限请求、安全/信任、失败重试、设置/开关、支付场景、成功通知、失败挽回、系统通知——每个模板含结构、信息递进、必须遵守的规则、3 种风格变体。

### 5 种风格预设

克制高级 / 强推促销 / 严谨合规 / 温暖陪伴 / 中性专业——场景决定风格，载体决定格式，两条轴独立组合。

### 心理学策略

框架效应、损失厌恶、顾虑消除、社交验证——每个策略标注适用场景和伦理红线。

## 快速示例

**输入**：帮我写一个支付失败的重试文案，弹窗场景

**输出**：

| 变体 | 标题 | 正文 | 按钮 |
|------|------|------|------|
| 中性专业 | 支付未完成 | 银行卡余额不足，更换支付方式或稍后重试 | 更换方式 / 重试 |
| 温暖陪伴 | 差一点点 | 可能是余额不太够，换个方式试试看 | 换个方式 / 再试试 |
| 严谨合规 | 交易未完成 | 您的银行卡余额不足，更换支付方式或稍后重试 | 更换方式 / 稍后重试 |

三个变体在正式度、温暖度、敬语、叠词四个语言维度上不同——不是换个说法，是换一套语言参数。

## 三种模式

| 模式 | 触发 | 产出 |
|------|------|------|
| **诊断** | "这段文案有什么问题" | 问题清单 + 严重度 + 风格定位 + 改写变体 |
| **改写** | "这个文案怎么改" | 先诊断再改写，2-3 组跨风格变体 |
| **生成** | "帮我写一个XX文案" | 场景路由 → 风格推断 → 2-3 组变体 |

## 使用方式

在 Claude Code 中直接描述文案需求：

- "帮我写一个错误提示文案" → 生成模式，自动推断参数
- "这段文案有什么问题" → 诊断模式
- "改一下这个按钮文案" → 改写模式
- "帮我写推送文案，限时优惠场景" → 生成模式（MKT 类型）
- "这个英文怎么翻成中文" → 改写模式 + 英中适配

Skill 会自动识别场景、载体、风格，输出 2-3 组变体。如果信息不足，会渐进式追问（最多 3 个问题）。

## 置信度说明

规则标注了置信度等级：

| 等级 | 含义 | 说明 |
|------|------|------|
| **高** | 有 A/B 测试数据或多源交叉验证 | 可以直接作为决策依据 |
| **中** | 有理论支撑和行业实践，缺中文 A/B 数据 | 建议参考方向，有条件时验证 |
| **待验证** | 基于理论推断 | 可作为参考方向，不建议作为决策唯一依据 |

**诚实边界**：大量核心发现缺中文 A/B 数据。有英文 A/B 数据的案例（Android Pay +12%、Google Hotel +17%、电商注册 +45%）会明确标注来源。参数值需实际产品测试校准。

## 伦理红线

1. **利益必须真实**——禁止虚构优惠、夸大效果
2. **损失框架仅限 MOFU/BOFU**——用户刚接触产品时禁用
3. **禁止 Confirmshaming**——退出选项不能含羞辱文案
4. **稀缺信息必须可验证**——"仅剩3件"须有真实库存
5. **社交验证数据必须真实**——禁止虚构

## 不适合的场景

品牌故事、公关稿件、英文文案、SEO 优化文——这些场景的规则体系与 UI 文案完全不同，本 Skill 不覆盖。
