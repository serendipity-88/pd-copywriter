# PD Copywriter

中文互联网产品文案引擎——让 AI 写出的每一句产品文案，都经过语言学和用户心理学的系统校验。

## 这个 Skill 解决什么问题？

写产品文案时，大多数团队靠"语感"和"审稿"。但语感不可复制，审稿没有标准。

结果是：

- **同一个 App 里，弹窗写"请确认"，按钮写"删除"，Toast 写"操作成功"**——三种语气，三种人称策略，三种标点习惯，用户感觉在和三个产品对话
- **"确定撤回吗？"的按钮是"确定/取消"**——用户不看标题就不知道点了会发生什么，这不是文案问题，是交互事故
- **"高达99.4%的成功率"对刚失败过的用户说**——惊喜语气对受挫用户是刺痛，不是鼓励
- **"3步才能完成"**——"才"让用户觉得门槛高；"3步就能完成"——"就"让用户觉得简单。同一个事实，一个虚词翻转了感知
- **错误提示写"参数丢失或错误"**——用户不知道发生了什么，也不知道该怎么办
- **"仅剩3件"永远挂在那里**——虚假的稀缺性在侵蚀用户信任

这些问题不是"语感不好"——它们是可以被系统识别、分类、修正的产品缺陷。PD Copywriter 把中文产品文案中反复出现的这些问题，拆解成可查询、可校验、可生成的规则体系。

## 没有它 vs 有它

| | 没有 PD Copywriter | 有 PD Copywriter |
|---|---|---|
| **一致性** | 同一产品内语气、人称、标点混乱，每个设计师各写各的 | 5 种风格预设 × 13 种场景模板，同一产品内语调统一 |
| **防错** | "确定/取消"按钮、虚假稀缺性、双重否定、中英标点混用——靠人工审稿发现 | P0 必查项自动拦截：按钮即动作、虚词框架正向性、空洞形容词、伦理红线 |
| **场景适配** | 空状态写"0张优惠券"，错误提示写"参数校验失败"，支付页写"立即购买！" | 13 种场景各有结构模板、信息递进规则、必须遵守的硬约束 |
| **风格选择** | "感觉这里应该温暖一点"——然后叠词和敬语混用 | FORMAL/WARMTH/FORCE 三维参数，变体之间在正式度/温暖度/信息密度/敬语/叠词/标点上有明确差异 |
| **载体约束** | Toast 写了 20 个字，用户还没读完就消失了 | 18 种 UI 组件的字数硬约束，从组件物理属性（字号、停留时间、可视区域）推算 |
| **跨类型差异** | Push 用 UI 的规则写（不用感叹号），规则用 UI 的规则写（不用句号） | UI/MKT/EXP 三类文案在标点、敬语、叠词、语气强度上规则完全不同，一键切换 |
| **心理学依据** | "感觉这样写转化好" | 框架效应、损失厌恶、禀赋效应、社交验证——每个策略标注适用场景和伦理边界 |

## 它怎么做到的？

三层引擎，从高可靠到高创造：

**Layer 1：查表引擎（99%+ 可靠）**——语法模板库、动词矩阵、载体字数约束、主语规则。这些是硬约束，违反就是错误。按钮必须写动作（"撤回/不撤回" > "确定/取消"），Toast 不能超过 8 个字，"删除"和"移除"必须匹配操作的不可逆性。

**Layer 2：推理引擎（~90%）**——虚词选择、叠词开关、英中适配、心理学策略。这些需要场景判断。"可以"在引导场景是友好邀请，在合规场景是冷硬规范；叠词在空状态降低挫败感，在金融场景削弱信任感。同一个词，场景不同，效果相反。

**Layer 3：韵律引擎（~70%）**——字数对称、冗余检测、节奏感。这些是锦上添花，不是雪中送炭。规则是：大声念一遍，不像人话就改。

## 7 个语言学子模块

| 模块 | 解决什么 | 举例 |
|------|---------|------|
| 语法模板库 | 正向/负向框架选择 | "3步就能完成" vs "3步才能完成"——同一个事实，感知完全不同 |
| 虚词选择引擎 | 近义词的语气差异 | "可"在 ToC 产品里 WARMTH=-2，"可以"正常——一个字的差异，推开用户还是邀请用户 |
| 动词选择矩阵 | 动词力度匹配不可逆性 | "删除"（不可逆）vs "移除"（可逆）vs "清理"（整理）——混用导致用户恐慌或误操作 |
| 主语选择规则 | 中文独有的情感开关 | 系统状态→省略主语（"已删除"），成功→"为你"（"已为你开启"），合规→"我们"（"我们将保护"） |
| 叠词开关 | 温暖度和能力度的 trade-off | 叠词 = WARMTH+2 但 FORCE-2——空状态用"试试"降低挫败，支付场景用叠词削弱信任 |
| 英→中适配 | 6 种系统性翻译失真 | Must→"需要"（不是"必须"），Can→按焦点区分（能力/许可/预测），Just→"只需/只要" |
| 韵律检查器 | 冗余、空洞、节奏 | "已经被删除了"→"已删除"，"很安全"→说出具体在哪，"高达99.4%"→对失败用户不真诚 |

## 三类文案，三套规则

中文产品文案不是一套规则走天下。UI 交互、营销推送、说明型文案在标点、敬语、叠词、语气强度上规则完全不同：

| | UI 交互 | 营销推送 | 说明型 |
|---|---|---|---|
| 句号 | 不用 | 不用 | **必须用** |
| 感叹号 | **禁用** | 利益场景克制 | **禁用** |
| "我" | **禁用** | 禁用 | 禁用 |
| "我们" | **禁用** | 谨慎 | **可用**（合规场景） |
| "请" | **禁用** | 可用 | 可用 |
| 叠词 | 空状态/引导可用 | 鼓励使用 | **禁用** |

这些不是偏好，是规则。用错了不是"风格不同"，是"类型错误"。

## 5 种风格预设 × 13 种场景模板

场景决定风格，载体决定格式——两条轴独立组合。

**5 种风格**：克制高级 / 强推促销 / 严谨合规 / 温暖陪伴 / 中性专业

**13 种场景**：错误提示、操作确认、空状态、功能引导、CTA/卖点、权限请求、安全/信任、失败重试、设置/开关、支付场景、成功通知、失败挽回、系统通知

每种场景有结构模板（标题/正文/按钮的字数约束和信息递进顺序）+ 必须遵守的硬约束 + 3 种跨风格变体。

## 心理学策略 + 伦理红线

框架效应、损失厌恶、禀赋效应、社交验证、顾虑消除——每个策略标注了适用场景和不适用场景。

5 条不可突破的伦理红线：利益必须真实、损失框架仅限 MOFU/BOFU、禁止 Confirmshaming、稀缺信息必须可验证、社交验证数据必须真实。

## 三种模式

| 模式 | 触发 | 产出 |
|------|------|------|
| **诊断** | "这段文案有什么问题" | 问题清单 + 严重度 + 风格定位 + 改写变体 |
| **改写** | "这个文案怎么改" | 先诊断再改写，2-3 组跨风格变体 |
| **生成** | "帮我写一个XX文案" | 场景路由 → 风格推断 → 2-3 组变体 |

信息不足时渐进式追问（最多 3 个问题），不会一次性暴露所有参数。

## 置信度说明

规则标注了置信度等级：

| 等级 | 含义 | 说明 |
|------|------|------|
| **高** | 有 A/B 测试数据或多源交叉验证 | 可以直接作为决策依据 |
| **中** | 有理论支撑和行业实践，缺中文 A/B 数据 | 建议参考方向，有条件时验证 |
| **待验证** | 基于理论推断 | 可作为参考方向，不建议作为决策唯一依据 |

**诚实边界**：中文产品文案领域严重缺少 A/B 测试数据。有英文 A/B 数据的案例（Android Pay +12%、Google Hotel +17%、电商注册 +45%、DiDi +287%）会明确标注来源。大量核心发现基于语言学理论和跨语言研究推演，参数值需实际产品测试校准。

## 安装

```bash
cd pd-copywriter
bash publish.sh
```

自动复制到 3 个安装目录：`~/.claude/skills/pd-copywriter`、`~/.codefuse/engine/cc/skills/pd-copywriter`、`~/.codefuse/fuse/skills/pd-copywriter`

手动安装：

```bash
mkdir -p ~/.claude/skills/pd-copywriter
cp -r skill/* ~/.claude/skills/pd-copywriter/
```

## 使用方式

在 Claude Code 中直接描述文案需求：

- "帮我写一个错误提示文案" → 生成模式，自动推断参数
- "这段文案有什么问题" → 诊断模式
- "改一下这个按钮文案" → 改写模式
- "帮我写推送文案，限时优惠场景" → 生成模式（MKT 类型）
- "这个英文怎么翻成中文" → 改写模式 + 英中适配

## 文件结构

```
skill/
├── SKILL.md                        ← 主 Skill 文件（自动注入上下文）
└── references/
    ├── grammar-templates.md         ← 语法模板库（正向框架、主语规则、错误提示等）
    ├── function-words-table.md      ← 虚词参数表（能愿动词、叠词、敬语、语气强度等）
    ├── verb-matrix.md               ← 动词选择矩阵（删除/创建/确认/取消类动词）
    ├── carrier-limits.md            ← 载体字数约束（18种UI组件的硬约束/软约束）
    ├── scene-templates.md           ← 场景级端到端模板（13个场景×3种变体）
    ├── flow-templates.md            ← 多步骤引导流程模板
    ├── style-presets.md             ← 5种风格预设×13场景跨风格表现
    ├── copy-style-gallery.md        ← 文案风格库（Apple/微信/拼多多等+A/B测试数据）
    ├── diagnostic-checklist.md      ← 诊断检查清单
    └── project-overrides.md         ← 项目覆盖规则模板
```

## 不适合的场景

品牌故事、公关稿件、英文文案、SEO 优化文——这些场景的规则体系与 UI 文案完全不同，本 Skill 不覆盖。
