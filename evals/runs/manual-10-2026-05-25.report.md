# Manual Eval Run: 10 Case Regression

Date: 2026-05-25

## Goal

Run a small but meaningful regression set after adding adjacent-copy guidance. The run checks whether PD Copywriter can keep P0 safety and clarity rules while also giving expert-level adjacent copy suggestions for single-slot requests.

This is a manual run using the current Skill instructions. It is not yet a blind model benchmark.

- 6 candidate cases from `evals/candidates.jsonl`

## What Worked

- P0 rules remained stable: button-as-action, negative framing, false scarcity, Toast compression.
- The new adjacent-copy rule worked on `pdw-candidate-006`: the output gave CTA variants, then added main title, subtitle, and CTA microcopy as optional adjacent positions.
- The output did not over-expand TapNow into a full page copy deck; it stayed within 1-3 adjacent suggestions.
- User-task naming worked well for settings labels: "管理设备", "设备更新", "通知设置", "数据与隐私".
- Payment coupon copy correctly separated benefit, threshold, card type, scope, and CTA destination.

## Observed Failure/Risk Patterns

No hard failures in this run, but several risks remain:

- The run is not blind. The same agent that updated the Skill generated the outputs, so this validates coherence more than independent generalization.
- The string-based scorer is brittle. It initially flagged a settings case because the output repeated bad original labels in an "原入口" column. The final recommendation was correct, but the scorer treated the repeated source text as forbidden output.
- Some diagnostic outputs are still longer than a busy PM may want. The Skill says to keep output concise, but actual model behavior may still over-explain.
- Multi-fragment cases like Pendo can pass traits while still needing deeper product-level rewrite judgment.
- The run has limited coverage of MKT and EXP generation. Most cases are UI-heavy.

## Updated Score

Previous working score: 8.2 / 10.

After this 10-case run: 8.4 / 10.

Reason for the increase:

- We now have evidence that the new soft/hard copy-slot rule does not break core P0 behavior.
- The TapNow-style "ask one, suggest adjacent two/three" behavior is present and controlled.
- Candidate eval coverage increased from structure-only to a runnable 10-case sample.

Why it is not higher:

- No independent blind run yet.
- No real installed Skill execution trace yet.
- Case library still lacks enough real-world Chinese app examples across major product types.
- Scoring is still mostly include/exclude based, not a validated LLM-as-judge or human rubric process.

## Next Best Step
