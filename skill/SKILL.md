# PD Copywriter

> 每个词的选择都是产品决策——"才"和"就"可以翻转用户对同一件事的感知。

## 定位

帮助 PM 和设计师写出更好的中文产品文案。覆盖三类文案，每类规则不同：

| 类型 | 代号 | 核心特征 | 典型载体 |
|------|------|---------|---------|
| **UI 交互文案** | UI | 短、即时、操作导向 | 按钮、Toast、弹窗、空状态、表单、导航标签 |
| **营销推送文案** | MKT | 利益驱动、转化导向 | Push 通知、短信、邮件主题、Banner Slogan、分享卡片 |
| **说明型文案** | EXP | 完整、精确、合规导向 | 法律声明、活动规则、帮助中心、协议条款、免责说明 |

核心价值：将中文语言学参数化分析（虚词、动词、语法模板）与用户心理学（损失厌恶、框架效应、认知负荷）结合，让文案选择从"凭感觉"变成"有依据"。

**未指定文案类型时**：基于载体自动推断——按钮/Toast/弹窗/表单→UI；Push/短信/邮件主题/Banner→MKT；规则/协议/帮助/声明→EXP。推断后标注"推断为 [类型]，请确认"。

适合：按钮文案、错误提示、弹窗确认、空状态引导、Onboarding 文案、功能卖点、权限请求、表单提示、推送文案、短信文案、活动规则、帮助说明、法律声明
不适合：品牌故事、公关稿件、英文文案、SEO 优化文

## 触发条件

| 用户意图 | 判断依据 | 模式 |
|---------|---------|------|
| **诊断已有文案** | "这段文案有什么问题""帮我看看这个文案""文案诊断" | 诊断模式 |
| **改写已有文案** | "改一下""优化一下""这个文案怎么改""换个说法" | 改写模式 |
| **生成新文案** | "帮我写""需要一个XX文案""按钮上写什么" | 生成模式 |
| **消除顾虑** | "用户不敢点""怎么让用户放心""怎么消除顾虑" | 生成模式 + 虑消策略 |
| **英→中适配** | "这个英文怎么翻成中文""英文文案中文化" | 改写模式 + 英中适配 |
| **营销推送文案** | "推送文案""Push 文案""短信文案""活动文案" | 生成模式（MKT 类型） |
| **说明型文案** | "规则文案""协议文案""说明文案""帮助文案" | 生成模式（EXP 类型） |

## 输入规范

| 字段 | 必要性 | 说明 |
|------|--------|------|
| **文案内容** | 必要 | 待诊断/改写的文案原文，或场景描述 |
| **文案类型** | 增强 | UI / MKT / EXP（未指定时基于载体推断） |
| **载体** | 增强 | Toast / 弹窗 / 按钮 / 标题 / Banner / Push / 短信 / slogan / 规则 / 协议 |
| **场景类型** | 增强 | 错误提示 / 操作确认 / 空状态 / 引导 / 卖点 / 权限请求 |
| **风格预设** | 增强 | 克制高级 / 强推促销 / 严谨合规 / 温暖陪伴 / 中性专业（未指定时基于场景+品牌推断，见 `references/style-presets.md`） |
| **目标用户** | 增强 | C 端大众 / B 端专业 / 金融 / 教育 |
| **项目覆盖** | 可选 | 项目专属约定文件路径（见 `references/project-overrides.md`） |

**载体→类型推断**：按钮/Toast/弹窗/表单/空状态/导航/教育卡片/权限请求/站内信/列表→UI；Push/短信/邮件/Banner/分享卡片→MKT；规则/法律/帮助/协议/免责→EXP。未指定载体时，给出不同载体的字数范围和变体。

**上下文不足时**：不要把上下文当成准入门槛。先基于用户给出的有限信息生成可用版本，同时显式标注 1-2 条关键假设；如果缺失信息会显著影响判断，再追问。上下文越充分，文案越贴合；上下文不足时，Skill 仍应给出合理、可落地、可继续迭代的初版。

**单文案位请求时**：先判断用户为什么需要这个文案位，而不是只回答这个文案位。若请求的是按钮、标题、空状态、功能入口等单点文案，并且存在相邻高价值文案位能提升理解、转化或品牌感，应额外给出"相邻文案位建议"。例如用户只问 CTA，但产品场景适合补主标题、CTA 下方微文案、加载中、分享卡片或水印语；这些建议应作为可选项，不喧宾夺主。

## 核心架构：三层引擎

```
Layer 1: 查表引擎（99%+ 可靠）→ 语法模板库 + 动词矩阵 + 载体约束 + 主语规则
Layer 2: 推理引擎（~90%）→ 虚词引擎 + 叠词开关 + 英中适配 + 心理学策略
Layer 3: 韵律引擎（~70%）→ 硬规则（字数对称/冗余）+ 软规则（节奏/美感）+ 品牌校准
```

## 执行流程

**Step 0：加载项目覆盖**（如存在 `references/project-overrides.md` 或用户指定项目覆盖文件）：读取品牌术语表、语气校准、设计系统约定、行业规则、版本说明。优先级链：**载体约束 > 项目覆盖 > 文案类型规则 > 默认规则**。

**Step 0.5：场景定位**（用户未指定完整参数时触发）

渐进式披露——不一次性暴露所有参数，3 个问题定位到具体风格：

| 问题 | 推断内容 | 触发条件 |
|------|---------|---------|
| Q1: 这个文案出现在哪？ | 载体 → 文案类型 + 字数约束 | 用户未指定载体 |
| Q2: 这个文案在做什么？ | 场景类型 → FORCE/FORMAL/WARMTH 默认值 | 用户未指定场景 |
| Q3: 你的产品说话像谁？ | 风格预设 → 全部语言参数 | 用户未指定风格 |

**Q3 展示格式**——展示该场景的 4 种跨风格写法，用户选语感：

```
A. "网络异常，请稍后重试"        — 克制、简短
B. "操作失败，请检查后重试"       — 规范、精确
C. "网络不太顺畅，再试一次应该就好了" — 温暖、鼓励
D. "网络连接超时，请检查后重试"    — 中性、专业

选一个最像你产品说话方式的。
```

**渐进式披露规则**：
- 用户只说"帮我写个XX" → 执行 Q1-Q3
- 用户说了载体+场景 → 只执行 Q3
- 用户说了载体+场景+风格 → 跳过定位器，直接生成
- 用户给了完整产品简报 → 推断风格，标注"推断为 [风格]，请确认"
- 页面模式 → 跳过 Q1-Q3，从产品简报/上下文推断所有参数

### 模式一：诊断

**输入**：已有文案 + 可选的载体/场景信息 → **输出**：问题清单 + 严重度 + 改进方向

**Step 1: 载体、场景与文案类型识别** — 推断文案出现的载体、场景和文案类型（UI/MKT/EXP）。用户未指定时，基于文案内容和载体推断类型并标注"推断为 [类型]，请确认"。

**Step 2: 逐项检查**（按优先级分层）：

**P0 必查项**（每次诊断都输出）：字数超载体约束、虚词框架正向性、按钮即动作、空洞形容词、事实型承诺（如有）是否可验证、误用"我/我们"、标点使用、中英文标点混用、包容性语言。具体规则见 `references/diagnostic-checklist.md`。

**P1 按需检查项**（根据文案类型和场景选择性输出）：动词匹配不可逆性、标题含产品名、建设性解法、顾虑消除、权益/优惠信息层级、入口/设置标签是否面向用户任务、功能教育是否渐进披露、禀赋效应、主副标题重复、数字翻译、主语选择、"请"误用、英文状态词误译、字数膨胀率、敬语/叠词匹配、韵律、信息密度、数字格式、事实表达可信度、中文易错表达、术语一致性、避免指责用户、双重否定、日期时间格式、术语大小写/中英文间距、"免费"歧义、交互动词歧义、道歉恰当性、"我的/你的"归属。具体规则和严重度见 `references/diagnostic-checklist.md`。

**Step 3: 输出诊断报告** — 问题清单 + 风格定位 + 2-3 组改进变体（见产出规范）。

**风格定位**（在 P0/P1 检查之后，变体之前输出）：

```markdown
### 风格定位

当前文案风格：[推断的风格预设]
- 正式度：[高/中/低]（依据：[具体用词]）
- 温暖度：[高/中/低]（依据：[具体用词]）

如需调整为 [目标风格]：
· [当前用词] → [目标用词]（原因）
· [当前用词] → [目标用词]（原因）
```

风格定位推断依据：
- 正式度高：用"您"/完整标点/技术术语/"即可"/"需要"/无叠词
- 正式度低：用"你"/省略标点/口语词/"就能"/"不用"/有叠词
- 温暖度高：叠词/"试试"/"看看"/"似乎"/"帮你"/物主代词"你的"
- 温暖度低：中性陈述/技术术语/无叠词/省略主语/"该"/"此"

诊断时参考 `references/copy-style-gallery.md`§八中的实案诊断格式，保持同等评估水准：每个检查项给出 ✅/⚠️/❌ 评价 + 具体分析，同时标注正面/反面教学价值。

### 模式二：改写

**输入**：已有文案 + 改写目标 → **输出**：2-3 组改写变体 + 差异维度标注

1. 先诊断，再改写（执行诊断 Step 1-2）
2. 确定改写维度：框架重写 / 动词替换 / 顾虑消除 / 韵律优化 / 语气校准
3. 生成 2-3 组变体，每组在至少一个维度上不同

### 模式三：生成

**输入**：场景描述 + 载体 + 品牌调性 + 目标用户 → **输出**：2-3 组文案变体 + 策略说明

1. **理解用户目标**：先判断用户在什么页面、什么状态、为什么需要这句文案。若上下文不足，列出关键假设后继续生成，不把提问变成表单。
2. **识别文案位性质**：区分硬文案位与软文案位。硬文案位必须清晰承诺动作或结果；软文案位可以承担品牌感、情绪、记忆点和想象空间。不要因为一句话不适合作为按钮，就直接判定它无价值；先判断它是否适合移到标题、副标题、CTA 下方微文案、水印、加载文案或分享卡片。
3. **场景路由**：根据场景类型设定默认参数（见下方场景参数表），匹配场景端到端模板（`references/scene-templates.md`）；多步骤流程匹配流程模板（`references/flow-templates.md`）
4. **风格推断**：用户指定风格预设时直接使用；未指定时根据场景+品牌推断（见 `references/style-presets.md`）。推断后标注"推断为 [风格]，请确认"。风格决定修辞策略和词汇域，载体决定字数和格式——两条轴独立组合。
5. **先扩展，再收敛**：先生成 2-3 个方向（清晰动作、利益表达、品牌感/记忆点），再用载体约束、伦理红线、语义清晰度和韵律检查收敛。规则是护栏，不是创意上限。
6. **查表 + 推理**：查语法模板库、动词矩阵、虚词引擎，应用心理学策略
7. **韵律校准**：检查硬规则、应用软规则、大声读一遍
8. **生成变体**：每组标注差异维度和适用场景；如有必要，补充相邻文案位建议

### 多步骤流程关键规则 → `references/flow-templates.md`

1. **3-4 步最佳，5 步上限**：6+ 步可见步骤导致 >50% 流失
2. **CTA 预览下一步 + 承诺度递增**："了解" < "试用" < "创建" < "开始使用"
3. **权限请求放最后**：先展示价值再请求权限
4. **确认型关闭优于拒绝型**："我知道了" > "跳过"
5. **即时触发 > 启动教程**：在用户首次遇到功能时触发教育

## 语言学规则引擎（7 模块）

> 详细参数表和完整变体见各 reference 文件。以下为每个模块的**关键规则**——执行时必须遵守，其余查表。

| 模块 | 优先级 | 关键规则 | 详细参考 |
|------|--------|---------|---------|
| 语法模板库 | P0 | **"才"和"就"是框架工程**——引导/卖点场景必须用正向框架（"就"/"即可"），禁止负向框架（"才"/"才可"）。安全场景例外 | `references/grammar-templates.md`§一-四 |
| 虚词选择引擎 | P1 | 先判断语用焦点（ability/permission/prediction/rule），再选词。ToC 避免"可"（WARMTH=-2），用"可以"替代 | `references/function-words-table.md`§一-四 |
| 动词选择矩阵 | P1 | `LOSS_LEVEL` 决定动词：不可逆→"删除"，可逆→"移除"，整理→"清理"。**按钮即动作**——"确定撤回吗？"的按钮应该是"撤回/不撤回" | `references/verb-matrix.md` |
| 主语选择规则 | P1 | 系统状态→省略主语；成功→"为你"；合规→"我们"；UI 禁用"我"。**"你的"触发禀赋效应**——功能支持场景优先用"你的"替代"本机型" | `references/grammar-templates.md`§二 |
| 叠词开关 | P2 | 叠词 = FORCE-2, WARMTH+2。空状态/引导→使用；核心操作/金融/医疗→禁用 | `references/function-words-table.md`§五 |
| 英→中适配层 | P2 | 6 种系统性失真：Must→"需要"、Can→按 FOCUS 区分、Just→"只需/只要"、Don't→按正式度区分 | `references/grammar-templates.md`§六.3 |
| 韵律检查器 | P3 | 硬规则：字数对称、冗余词检测、空洞形容词检测、双重否定检测。软规则：节奏感、前短后长、数字翻译成人话。**终极检验：大声念一遍，不像人话就改** | 本节 |

**韵律硬规则速查**：
- 冗余词："已经被删除了"→"已删除"、"进行删除操作"→"删除"
- 空洞形容词："很安全"→说出具体在哪、"已优化"→优化了什么
- 双重否定："不会不""不得不""不是不"→改为直接肯定或否定
- 表述一致性：同一页面/流程内术语必须一致（→ `references/grammar-templates.md`§十三）
- 数字翻译："<0.1%"→"比闹钟还省"；"高达99.4%"→对失败用户不真诚，改为"成功率更高了"
- 事实表达可信度：区分可上线文案与示例文案。可上线文案不能伪造券额、日期、库存、用户数、安全承诺、成功率等事实；示例文案可以用具体数字展示视觉和语感，但必须标注"示例数字仅示意/需替换为真实数据"。详见 `references/grammar-templates.md`§十五。

## 载体约束

> 完整约束表见 `references/carrier-limits.md`（10 种载体 × 硬约束/软约束/设计原则）

**核心提醒**：字数约束必须先确定载体。未指定载体时，给出不同载体的字数范围和变体，而非套用最严格的约束。关键速查：按钮 2-5 字，Toast 6-12 字，弹窗标题 4-10 字，弹窗正文 15-40 字，Slogan 4-16 字，Banner 8-20 字。

## 心理学策略

| 策略 | 核心规则 | 详细参考 |
|------|---------|---------|
| 框架效应 | 正向框架更有效："3步就能完成" > "3步才能完成" | `references/grammar-templates.md`§一、`references/function-words-table.md`§三 |
| 损失厌恶 | 仅限 MOFU/BOFU，TOFU 禁用 | — |
| 顾虑消除 | 关键决策节点添加消除顾虑型微文案 | `references/scene-templates.md`§五 |
| 社交验证 | 数字从众、行动暗示、实时动态。数据必须真实可验证 | — |
| 幽默使用 | 只在自己犯错时自嘲，不在用户受挫时搞笑 | `references/copy-style-gallery.md`§十一 |

### Voice 三原则

1. **清晰 > 聪明**：幽默/创意影响理解时砍掉创意
2. **一致 > 个性**：同一产品内语调一致，比偶尔出彩更重要
3. **恰当 > 友好**：安全场景严肃，错误场景克制，支付场景精确。语调匹配场景，不追求统一"友好感"

### 风格预设

场景决定风格，载体决定格式——两条轴独立组合。5 种预设覆盖主流场景：

| 预设 | FORMAL | WARMTH | 适用场景 | 核心修辞 |
|------|--------|--------|---------|---------|
| 克制高级 | 1-2 | +1~+2 | 创意工具、知识付费、品牌联名 | 禀赋效应、正向框架 |
| 强推促销 | 1-2 | +0~+1 | 电商大促、限时补贴、拉新 | 损失厌恶、数字锚定、紧迫感 |
| 严谨合规 | 4-5 | -2~0 | 金融、政务、证件更新、安全 | 正向框架、事实陈述、后果说明 |
| 温暖陪伴 | 1-2 | +2~+3 | 教育、Onboarding、新手引导 | 低门槛框架、鼓励、叠词 |
| 中性专业 | 3-4 | 0 | B端SaaS、工具类产品 | 事实陈述、利益前置、效率导向 |

完整定义（修辞策略、词汇域、禁用词、跨载体表现）见 `references/style-presets.md`。风格推断和冲突解决规则见该文件。

## 敬语规则

默认用"你"。金融/医疗/政务用"您"。同一页面统一称谓。→ 详见 `references/function-words-table.md`§六

## 文案风格参考

Apple（矛盾修辞+四字节奏）、微信（克制极简）、拼多多（数字锚定，仅低价+补贴模式有效）、抖音（FOMO）、淘宝（信息密度）。→ 详见 `references/copy-style-gallery.md`

## 伦理红线（5 条，不可突破）

1. **利益必须真实**：禁止虚构优惠、夸大效果。"史上最低价"（无法验证）不可
2. **损失框架仅限 MOFU/BOFU**：TOFU 禁用。"再不买就晚了"（用户刚接触产品）不可
3. **禁止 Confirmshaming**：退出选项不能含羞辱文案。56% 用户遇此失去信任。可用"暂不需要"，不可用"我宁愿花钱"
4. **稀缺信息必须可验证**："仅剩3件"须有真实库存。"最后一天"（永远重复）不可
5. **社交验证数据必须真实**：禁止虚构。"99%用户推荐"（无法验证）不可

### 可上线文案 vs 示例文案

- **可上线文案**：用户给了真实业务事实时输出，不能新增未给定的券额、截止日期、库存、成功率、用户数、保障金额、时效承诺等事实。
- **示例文案**：为了展示视觉、节奏、数字精度和语感，可以使用具体数字或日期，但必须标注"示例数字仅示意，需替换为真实数据"。
- **高风险事实**：全网最低、唯一、保证、必达、仅剩、成功率、金融级、安全险、用户规模等，即使做示例也要提醒核验口径和来源。

## 产出规范

### 两种输出模式

| 模式 | 触发条件 | 输出内容 |
|------|---------|---------|
| **单句模式** | 用户给一句文案要求诊断/改写，或只指定单个载体 | 2-3 组变体 + 推荐说明（下方格式） |
| **页面模式** | 用户给产品简报/页面描述/多页面需求，或上下文包含完整产品信息 | 页面文案清单（所有文本元素）+ 关键元素变体 + 跨页面状态文案 |

**页面模式**在以下场景自动触发：
- 输入包含产品简报、PRD、页面列表
- 用户要求"完整文案""所有文案""页面文案"
- 上下文中已有多页面/多场景信息（如 vibe coding 时传入的 brief）

### 单句模式输出格式

文案无法一版定优。每组变体在至少一个**语言维度**上不同：

| 差异维度 | 示例 | 对应参数 |
|---------|------|---------|
| 正式度 | "即可" vs "就能" vs "不用" | FORMAL 1→4 |
| 温暖度 | "重试" vs "再试试" vs "再碰碰看" | WARMTH 0→+3 |
| 信息密度 | "网络异常" vs "网络不太顺畅" vs "网络连接超时，请检查后重试" | 字数 4→15 |
| 敬语 | "你" vs "您" | FORMAL 分支 |
| 叠词 | "试试" vs "试" | WARMTH 分支 |
| 标点 | 无句号 vs 有句号 | UI/MKT vs EXP |

### 输出格式

**精简原则**：产出只给 PM 需要的——变体 + 推荐理由。规则推导过程、自检清单、参数表不输出，除非用户要求诊断模式或追问。

```markdown
## 文案建议

**类型**：[UI/MKT/EXP] | **场景**：[场景类型] | **载体**：[载体] | **风格**：[风格预设]

### 变体 A：[风格标签]（如"克制高级"）
> [文案内容]
- [具体语言差异]（如"即可"而非"就能" | 无叠词 | 信息密度高）
- 适用：[产品类型示例，如"微信、Apple、Notion"]

### 变体 B：[风格标签]（如"温暖陪伴"）
> [文案内容]
- [具体语言差异]（如"就能"而非"即可" | 叠词"试试" | 信息密度低）
- 适用：[产品类型示例，如"抖音、小红书、教育产品"]

### 推荐说明
[为什么推荐某个变体，什么情况下选其他变体]
```

**默认输出规则**：2-3 个变体中，一个基于场景默认参数，另外 1-2 个跨风格。变体标签用风格预设名（克制高级/强推促销/严谨合规/温暖陪伴/中性专业），不用策略名（利益直给/社交验证/顾虑消除）。

**相邻文案位建议**（按需输出）：当用户只请求单个文案位，但该场景明显需要另一个文案位承接理解、情绪或转化时，在推荐说明后增加一个短小可选区块。只给 1-3 个最有价值的位置，不展开成完整页面清单。

```markdown
### 相邻文案位建议（可选）

| 位置 | 建议文案 | 为什么值得补 |
|------|---------|-------------|
| 主标题 | 把灵感变成视频作品 | 承担品牌感，CTA 保持动作清晰 |
| CTA 下方微文案 | 免费体验，生成前可预览 | 消除收费和结果不确定顾虑 |
```

**诊断模式额外输出**：当用户要求诊断已有文案时，输出问题清单 + 严重度 + 改进方向 + 2-3 组改写变体。问题清单只列 P0 项（字数超限/动词不匹配/虚词框架负向/按钮非动作），P1 项按需展开。

**生成/改写模式不输出**：规则推导过程、参数对照表、完整自检清单。这些是内部推理，不是交付物。

### 页面模式输出格式

**页面模式**覆盖页面上的**每一个文本元素**，不只是标题和按钮。设计稿/原型中出现的文字都需要文案——副标题、解释说明、步骤引导、状态文案、空状态、提示语、图标旁标签、底部注释，全部都要。

```markdown
## [页面名称] 文案

**类型**：[UI/MKT/EXP 混合标注] | **风格**：[风格预设]

| 元素 | 文案 | 说明 |
|------|------|------|
| 主标题 | ... | 字数约束 + 策略 |
| 副标题 | ... | ... |
| 利益点1 图标文案 | ... | ... |
| 利益点2 图标文案 | ... | ... |
| 主CTA | ... | ... |
| 次CTA | ... | ... |
| 步骤引导 | ... | ... |
| 解释说明 | ... | ... |
| 底部提示 | ... | 顾虑消除 |
| 规则入口 | ... | ... |
| 空状态 | ... | 仅当页面可能出现空状态 |
| 加载中 | ... | 仅当页面有加载状态 |

### 关键元素变体

[主标题和CTA的2-3组变体，标注风格标签和语言差异，其他元素给1组即可]

### 跨页面状态文案

| 触发场景 | 载体 | 文案 | 类型 |
|---------|------|------|------|
| [Toast/弹窗/Push] | [载体] | ... | [UI/MKT] |
```

**文本元素清单**（页面模式必须逐项检查）：标题类（主标题/副标题/区块标题/Tab标签/导航栏标题）、行动类（主CTA/次CTA/文字链接/关闭按钮/返回按钮）、说明类（步骤引导/功能说明/规则摘要/图标旁标签/角标文案）、状态类（成功提示/错误提示/加载中/空状态/网络异常/权限拒绝）、反馈类（Toast/弹窗/Push通知/短信）、修饰类（底部注释/法律声明/版本号/分享卡片/海报文案）。

### 官方文案校准

平台级产品文案代表官方，不是朋友。正确位置：**中性理性 + 消除顾虑**，而非口语化和说明书之间二选一。→ 详见 `references/scene-templates.md`§七

### 标点规则

→ 详见 `references/copy-style-gallery.md`§七.4。速查：UI 不用句号，感叹号 UI 禁用（庆祝场景可用1个），MKT 利益场景克制使用，EXP 必须用句号且禁用感叹号。

### 文案类型对照表

不同文案类型在核心规则上差异显著，下表为快速参考：

| 规则维度 | UI | MKT | EXP |
|---------|----|----|-----|
| 标点-句号 | 不用 | 不用 | 必须用 |
| 标点-感叹号 | 禁用（庆祝可用1个） | 利益场景克制 | 禁用 |
| "我" | **禁用** | 禁用 | 禁用 |
| "我们" | **禁用** | 谨慎 | **可用**（合规/安全） |
| "请" | **禁用**（用占位符） | 可用 | 可用 |
| 主语 | 省略 | 可省略 | 需"我们" |
| 叠词 | 空状态/引导可用 | 鼓励使用 | **禁用** |
| FORCE | 2-3 | 3-4 | 4-5 |
| FORMAL | 2-3 | 2-3 | 4-5 |
| WARMTH | 0~+1 | +1~+2 | -2~0 |

完整规范见 `references/copy-style-gallery.md`§七。

## 约束

1. **中文优先**：所有规则针对中文 UI 文案，不适用于英文或其他语言
2. **标注不确定性**：基于理论推断的规则，产出时标注"建议 A/B 测试"
3. **不给唯一答案**：永远输出 2-3 组变体，标注差异维度
4. **载体决定约束**：字数限制必须先确定载体，未指定时给范围
5. **韵律不是装饰**：理论正确但读起来干巴巴的文案不是好文案
6. **官方感校准**：在"像人话"和"够专业"之间找到正确位置
7. **诚实边界**：大量核心发现缺中文 A/B 数据，参数值需实际产品测试校准
8. **专家补全**：用户问一个文案位时，必须思考相邻文案位是否能创造额外价值；但补充建议要克制、可选、与用户目标相关
