# 场景级端到端模板

> 每个模板是完整文案的组装图——结构、信息递进、关键规则、变体示例。
> 所有示例用占位符（`[产品名]` `[功能]` `[核心利益]`），适用于任何产品。
> 碰一下、抖音、支付宝等真实产品案例放在"参考案例"中。

## 一、错误提示

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 问题 | 4-10 字 | 说清出了什么问题，用用户理解的语汇 |
| 原因（可选） | 5-15 字 | 只在帮助用户理解时添加，不暴露技术细节 |
| 解法 | 按钮 2-5 字 + 可选微文案 5-15 字 | 至少一个行动出口，不让流程断掉 |

### 完整文本元素清单

页面模式下，错误提示可能出现的所有文本元素：

| 元素 | 字数 | 必要性 | 说明 |
|------|------|--------|------|
| 错误标题 | 4-10 字 | 必要 | 弹窗标题，说清问题 |
| 错误正文/原因 | 5-15 字 | 增强 | 只在帮助理解时添加 |
| 主操作按钮 | 2-5 字 | 必要 | 具体动词，"重试""重新提交" |
| 次操作按钮 | 2-5 字 | 可选 | "返回""取消" |
| 辅助链接 | 2-6 字 | 可选 | "忘记密码？""联系客服" |
| Toast 错误文案 | 6-12 字 | 可选 | 非弹窗场景的简短提示 |
| 错误页大标题 | 4-8 字 | 可选 | 全屏错误页的标题 |
| 错误页说明 | 10-25 字 | 可选 | 全屏错误页的详细说明 |
| 错误页插图旁文案 | 4-10 字 | 可选 | 插图下方的轻量说明 |

### 信息递进

```
问题 → [原因] → 解法
```

先让用户知道"怎么了"，再告诉"怎么办"。不要反过来。

### 必须遵守的规则

1. **用用户语汇，不用技术术语**："密码错误" > "参数校验失败" → `grammar-templates.md`§六.2
2. **每个错误至少一个行动按钮**：never 只给"好的" → `grammar-templates.md`§六.2
3. **正面表述 + 减轻挫败**：→ `grammar-templates.md`§六.2
4. **不暴露技术细节**："暂无内容" > "参数丢失或错误" → `grammar-templates.md`§六.2

### 变体示例

**变体 A：简洁型**（Toast / 小弹窗）

示例：密码错误 → [再试一次] [重置密码]

语言特征：FORMAL=3-4, WARMTH=0
· 无叠词、无敬语、中性陈述
· "密码错误"而非"密码似乎不对"——事实陈述，不加软化

**变体 B：有帮助型**（弹窗 / 错误页）

示例：支付未完成 → 网络不稳定，检查后重试 → [再试一次]

语言特征：FORMAL=3, WARMTH=+1
· 增加原因说明——信息密度更高
· "检查后重试"——中性指导，不使用敬语"请"（UI 禁用"请"，见 grammar-templates.md§二）

**变体 C：温暖型**（品牌调性轻松的产品）

示例：密码似乎不对 → [再试一次] [重置密码？]

语言特征：FORMAL=1-2, WARMTH=+2
· "似乎"软化语气——降低挫败感
· "再试一次"——叠词节奏感，行动感+温暖

→ 更多案例见 copy-style-gallery.md

## 二、操作确认

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 标题 | 4-10 字 | 明确操作对象，"确定撤回这条消息吗？" > "确定撤回吗？" |
| 正文（可选） | 10-25 字 | 说明操作后果，尤其是不可逆操作 |
| 按钮组 | 各 2-5 字 | 具体动词，"撤回/不撤回" > "确定/取消"；不可逆操作按钮可 6-8 字（→ `carrier-limits.md`§一） |

### 信息递进

```
操作对象 → [后果预告] → 具体动词按钮
```

### 必须遵守的规则

1. **按钮即动作**："撤回/不撤回" > "确定/取消"——不看标题就能判断操作结果（→ `verb-matrix.md`）
2. **不可逆操作预告后果**："删除后无法恢复"——让用户知情，而非用强词吓人
3. **破坏性操作用强动词+红色**："删除"而非"移除"——动词力度匹配不可逆性
4. **标题包含操作对象**："确定删除这张照片吗？" > "确定删除吗？"

### 变体示例

**变体 A：非破坏性确认**（可逆操作）

示例：移除这个成员？ → [移除] / [算了]

语言特征：FORMAL=2, WARMTH=+1
· "移除"而非"删除"——轻动词，暗示可逆
· "算了"而非"取消"——口语化退出，降低焦虑

**变体 B：破坏性确认**（不可逆操作）

示例：删除这个项目？ → 删除后无法恢复 → [删除] / [返回]

语言特征：FORMAL=4, WARMTH=-1
· "删除"强动词——力度匹配不可逆性
· "无法恢复"——中性陈述后果，不用"永远"等情绪词

**变体 C：替换确认**（有替代方案）

示例：退出编辑？ → 修改尚未保存 → [不保存] / [继续编辑]

语言特征：FORMAL=3, WARMTH=0
· "继续编辑"——正向引导替代"取消退出"
· 按钮即动作——"不保存/继续编辑" > "确定/取消"

→ 更多案例见 copy-style-gallery.md 及 `verb-matrix.md` 确认类

## 三、空状态

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 标题 | 4-8 字 | 点明当前状态，"暂无优惠券" > "0张优惠券" |
| 正文 | 10-25 字 | 给行动建议或解释原因 |
| 按钮（可选） | 2-5 字 | 引导下一步操作 |

### 信息递进

```
当前状态 → [安抚/解释] → 行动建议
```

### 必须遵守的规则

1. **不用"0"**："暂无优惠券" > "0张优惠券"——0 是边缘值，不符合语言习惯
2. **用叠词/温暖语调**："试试搜索其他内容"——WARMTH+2，降低挫败感
3. **始终提供行动出口**：空状态不能是死胡同，至少一个按钮或链接 → `grammar-templates.md`§六.2
4. **标题点状态，正文给方向**：标题说"是什么"，正文说"可以怎么做"

### 变体示例

**变体 A：引导行动型**

示例：暂无订单 → 去逛逛，发现好东西 → [去逛逛]

语言特征：FORMAL=2, WARMTH=+1
· "暂无"而非"0"——避免边缘值，中性陈述
· 叠词CTA"逛逛"——WARMTH+2，降低空状态挫败感

**变体 B：解释安抚型**（非用户原因的空状态）

示例：暂无消息 → 还没有人给你发消息，主动打个招呼吧 → [发消息]

语言特征：FORMAL=2, WARMTH=+2
· "还没有人"——暗示未来会有，不是永久的空
· "主动打个招呼吧"——叠词节奏+鼓励语气

**变体 C：趣味型**（品牌调性活泼的产品）

示例：这里空空如也 → 去看看有什么好玩的 → [去看看]

语言特征：FORMAL=1, WARMTH=+3
· "空空如也"——叠词化成语，趣味感
· "去看看"——叠词CTA，行动感+轻松

→ 更多案例见 copy-style-gallery.md

## 四、功能引导

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 主标题 | 6-15 字 | 核心利益 + 产品/功能名 |
| 副标题 | 10-25 字 | 补充信息（怎么做 / 社交验证 / 顾虑消除），不重复主标题 |
| CTA | 2-5 字 | 具体行动引导 |

### 完整文本元素清单

页面模式下，功能引导页面可能出现的所有文本元素：

| 元素 | 字数 | 必要性 | 说明 |
|------|------|--------|------|
| 导航栏标题 | 2-6 字 | 必要 | 功能名或"新功能" |
| 主标题 | 6-15 字 | 必要 | 核心利益 + 功能名 |
| 副标题 | 10-25 字 | 必要 | 补充信息，不重复主标题 |
| 利益点图标文案 | 2-6 字/条 | 增强 | 每个利益点一行，名词短语 |
| 利益点说明 | 5-15 字/条 | 增强 | 利益点的简短解释 |
| 步骤引导 | 5-15 字/步 | 增强 | "① 做什么 → ② 做什么 → ③ 结果" |
| 顾虑消除微文案 | 5-15 字 | 增强 | CTA 下方，如"你不会被收费" |
| 主CTA | 2-5 字 | 必要 | 具体行动引导 |
| 次CTA | 2-5 字 | 可选 | "暂不需要"等低承诺退出 |
| 底部提示 | 5-15 字 | 可选 | 法律声明、版本说明等 |
| 关闭按钮 | — | 可选 | 不加文案，× 即可 |
| 已完成状态 | 4-10 字 | 可选 | "已开启""已支持"等 |

### 信息递进

```
核心利益 → 补充信息（怎么做/谁在用/消除顾虑）→ 行动引导
```

主标题说"为什么好"，副标题说"怎么用"或"谁在用"，CTA 说"下一步做什么"。**主副标题信息不重复**。

### 必须遵守的规则

1. **主标题必须含产品/功能名**：用户只看标题要能知道这是什么
2. **正向框架**："就能" > "才能" → `grammar-templates.md`§四
3. **主副标题信息不重复**："很简单"+"3步搞定"重复了——副标题改说"怎么做"
4. **禀赋效应**："你的手机已支持" > "本机型已支持" → `grammar-templates.md`§十七
5. **CTA 用具体动词**："去试试" > "确定" → `verb-matrix.md`

### 变体示例

**变体 A：利益驱动型**（核心利益前置）

示例：[功能名]，解锁就能用 → 3步搞定，3亿人都在用 → [去试试]

语言特征：FORMAL=2, WARMTH=+1
· "就能"正向框架——低门槛暗示
· "去试试"——叠词CTA，降低启动门槛

**变体 B：社交验证型**（从众前置）

示例：3亿人都在用的[功能名] → 解锁就能用，[核心利益] → [我也试试]

语言特征：FORMAL=2, WARMTH=+1
· "3亿人"——数字锚定+社交验证
· "我也试试"——"也"暗示从众，叠词降低承诺感

**变体 C：禀赋效应型**（归属感前置）

示例：你的手机已支持 → [功能名]，解锁就能用 → [去试试]

语言特征：FORMAL=2, WARMTH=+2
· "你的"——物主代词触发禀赋效应
· "已支持"——省略主语+"已"客观陈述，暗示理所当然

### 参考案例

- 碰一下："解锁手机就能付，不用打开支付宝"——"就能"正向框架 + 利益前置（支付宝碰一下支付实测）
- 碰一下："你的手机已支持"——禀赋效应（支付宝碰一下支付实测）
- Android Pay："开始使用" > "添加卡片"（+12% CTR）——CTA 聚焦用户目标
→ 更多案例见 copy-style-gallery.md

## 五、CTA / 卖点

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 核心利益 | 6-15 字 | 正向框架，聚焦用户获得什么 |
| 顾虑消除（可选） | 5-15 字 | 消除用户最大的一个顾虑 |
| CTA | 2-5 字 | 低承诺动词 + 行动引导 |

### 硬文案位与软文案位

CTA 场景最容易把所有表达都塞进按钮。先判断文案位性质，再决定一句话应该放在哪里。

| 文案位 | 性质 | 应承担什么 | 不应承担什么 |
|--------|------|------------|--------------|
| 主CTA | 硬文案 | 明确动作、点击结果、承诺度 | 品牌 slogan、抽象情绪、复杂利益点 |
| 次CTA | 硬文案 | 退出、稍后、了解详情等低压力路径 | 羞辱式拒绝、模糊确认 |
| 主标题/卖点标题 | 软文案 | 产品价值、品牌感、记忆点 | 过长规则、按钮动作 |
| 副标题/补充说明 | 软文案 | 解释怎么做、为什么值得试 | 重复主标题 |
| CTA 下方微文案 | 软文案 | 消除收费、风险、门槛等顾虑 | 新增复杂条件 |
| 水印/分享卡片/加载文案 | 软文案 | 氛围、记忆点、传播感 | 关键操作承诺 |

**选择规则**：
- 如果一句文案有画面感但动作不清，不要强塞进按钮；优先考虑主标题、CTA 下方微文案、加载文案、水印或分享卡片。
- 如果用户只问 CTA，而产品是创意工具、内容生成、游戏、社区、消费决策等高感知场景，应主动给 1-3 条相邻文案位建议。
- 如果场景是支付、权限、删除、合规、错误恢复等高风险操作，优先保证硬文案清晰；相邻软文案只用于解释或消除顾虑，不做情绪渲染。
- 软文案可以打开想象空间，硬文案必须收束行动意图。两者组合，比把按钮写得既诗意又含糊更好。

### 完整文本元素清单

页面模式下，CTA/卖点页面可能出现的所有文本元素：

| 元素 | 字数 | 必要性 | 说明 |
|------|------|--------|------|
| 主标题/核心利益 | 6-15 字 | 必要 | 正向框架，聚焦用户获得什么 |
| 副标题/补充说明 | 10-25 字 | 增强 | 利益展开或顾虑消除 |
| 价格/数字锚定 | 2-8 字 | 增强 | "¥99""最高10元""3亿人都在用" |
| 顾虑消除微文案 | 5-15 字 | 增强 | CTA 下方，如"随时可取消""你不会被收费" |
| 主CTA | 2-5 字 | 必要 | 低承诺动词 + 行动引导 |
| 次CTA/退出文案 | 2-5 字 | 可选 | "暂不需要"（禁止 confirmshaming） |
| 利益点列表 | 3-8 字/条 | 增强 | 图标+短文案，每条一个利益 |
| 社交验证 | 5-15 字 | 增强 | "3亿人都在用""已有1000人参与" |
| 底部法律声明 | 5-20 字 | 可选 | "活动规则""隐私政策"等 |
| 倒计时/紧迫感 | 3-8 字 | MKT增强 | "仅剩3天""限时优惠"（数据必须真实） |

### 信息递进

```
核心利益 → [顾虑消除] → 低门槛行动
```

### 权益/优惠信息层级

适用于优惠券、会员权益、支付立减、包邮、积分兑换、新人礼包、试用权益等场景。不要为每一种权益单独建规则，先按信息层级拆解。

| 信息 | 作用 | 常见问题 | 示例 |
|------|------|----------|------|
| 利益金额/权益 | 用户获得什么 | "优惠"太空，未说金额或权益 | "最高减 5 元" |
| 使用门槛 | 什么时候可用 | 门槛和优惠上限混在一起 | "满 5 元可用" |
| 适用对象 | 谁/什么可用 | 卡、会员、商品、人群没说清 | "工商银行储蓄卡" |
| 适用场景 | 在哪里可用 | "通用（部分除外）"自相矛盾 | "部分支付场景可用" |
| 有效期 | 什么时候失效 | 紧迫感无依据 | "5月31日前可用" |
| 例外/规则 | 有什么限制 | 例外藏得太深，损害信任 | "具体以支付页为准" |
| CTA 去向 | 点后发生什么 | "去看看"不说明是查看规则还是使用 | "查看可用场景" / "去使用" |

**选择规则**：
- 用户还在了解权益 → CTA 用"查看权益""查看可用场景""了解规则"。
- 用户已在支付/下单链路 → CTA 用"去使用""立即使用""选择该优惠"。
- 规则复杂时，不把所有限制塞进卡片；卡片说核心条件，入口写"查看规则"。
- 金额表达去系统字段化："5.0元"→"5 元"或"¥5"；中文营销/支付文案避免无意义小数。
- 避免自相矛盾："通用场景（部分场景除外）"→"部分场景可用"或"多数支付场景可用，具体以页面为准"。

### 必须遵守的规则

1. **CTA 聚焦用户目标，非技术流程**："开始使用" > "添加卡片" → `grammar-templates.md`§七
2. **顾虑消除型微文案**：在 CTA 下方或旁边添加"你不会被收费""无需注册""随时可取消"
3. **低承诺动词降低决策重量**："了解" < "购买" → `grammar-templates.md`§七
4. **数字锚定利益**："省30%" > "更便宜" → SKILL.md 心理学策略
5. **禁止 confirmshaming**：退出选项用"暂不需要" → SKILL.md 伦理红线
6. **按钮不承载 slogan**：按钮写动作，标题/微文案/水印写品牌感。用户只问按钮时，也要判断是否需要补一个软文案位。

### 变体示例

**变体 A：利益直接型**

示例：限时5折，[产品名]年度优惠 → [立即查看]

语言特征：FORMAL=2, WARMTH=+1, FORCE=3
· 数字前置——"5折"放在最前面
· "限时"紧迫感——FORCE驱动，非WARMTH

**变体 B：顾虑消除型**

示例：免手续费转账 → 随时可撤回 → [了解详情]

语言特征：FORMAL=3, WARMTH=0, FORCE=1
· "随时可撤回"——顾虑消除，降低决策重量
· "了解详情"——低承诺CTA，"了解" < "购买"

**变体 C：社交验证型**

示例：3亿人都在用，[功能名] → [我也试试]

语言特征：FORMAL=2, WARMTH=+1, FORCE=2
· "3亿人"——数字锚定+从众暗示
· "我也试试"——"也"从众+"试试"低门槛

### 创意工具示例：单个 CTA 请求的专家补全

用户请求："TapNow 这个 AI 视频生成工具，首页 CTA 写什么？"

不要只给按钮。先拆文案位：

## 六、权限请求

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 标题 | 5-10 字 | 说明需要什么权限 |
| 正文 | 15-30 字 | 解释**为什么**需要 + 用户获得什么 |
| 按钮 | 2-5 字 | 允许/拒绝，或更具体的动词 |

### 信息递进

```
需要什么权限 → 为什么需要（使用价值）→ 允许/拒绝
```

### 必须遵守的规则

1. **必须解释"为什么"**：用户未知权限的目的远比权限本身更重要 → `grammar-templates.md`§五
2. **正面表述使用价值**："需要获取位置，帮你规划路线" > "需要获取位置信息"
3. **不说"必须""强制"**：用"需要"而非"必须"，除非是安全合规场景
4. **权限名称用用户理解的词**："位置" > "GPS"，"相机" > "摄像头权限"

### 变体示例

**变体 A：价值说明型**

示例：[产品名]需要位置权限 → 帮你找到附近的服务 → [允许] / [以后再说]

语言特征：FORMAL=3, WARMTH=+1
· "帮你"——价值导向，说明用户获得什么
· "以后再说"——低压力拒绝选项，比"拒绝"温暖

**变体 B：功能触发型**

示例：视频通话需要相机和麦克风 → 让对方看到和听到你 → [允许] / [拒绝]

语言特征：FORMAL=4, WARMTH=0
· "需要"——中性需求陈述，非"必须"
· "允许/拒绝"——标准正式选项，权限场景需要明确

### 参考案例

- 腾讯地图："需要获取您的位置信息" → "需要获取您的位置，帮您规划最优路线"（腾讯地图实测）
→ 更多案例见 copy-style-gallery.md

## 七、安全 / 信任

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 标题 | 6-15 字 | 正向框架：保障什么，而非防止什么 |
| 正文 | 15-30 字 | 具体保障措施 + 数字/背书 |
| CTA | 2-5 字 | 查看保障/了解详情 |

### 信息递进

```
保障什么 → 怎么保障（具体措施+数字）→ 了解更多
```

### 必须遵守的规则

1. **正向框架表达安全**："守护每一笔消费" > "防止盗刷" → `grammar-templates.md`§四
2. **不用"才能"表达保障**："解锁手机就能付" > "解锁手机才能付" → `grammar-templates.md`§四
3. **具体措施 > 空洞形容词**："不解锁付不了 + 全额赔付" > "很安全" → SKILL.md 韵律检查
4. **数字锚定安全感**："保额500万" > "高额保障" → SKILL.md 心理学策略
5. **官方感校准**：中性理性 + 消除顾虑，不是口语化也不是说明书 → SKILL.md 官方文案校准
6. **安全文案一次只说一个具体事实**：每个顾虑用一句具体机制消除，不用模糊短句覆盖多个顾虑。"别人碰不了"同时暗示"功能不可用"和"钱不会被偷"但两个都没说清——应拆成："不解锁付不了"（说机制）+ "每笔支付都会确认"（说流程），而非一个"别人碰不了"（模糊）

### 变体示例

**变体 A：守护型**（强调主动保护）

示例：[产品名]守护每一笔消费 → 免费百万安全险 + 多重安全防护 → [查看保障]

语言特征：FORMAL=3, WARMTH=+1
· "守护"而非"保护"——拟人化动词，温暖感+1，比"保护"更有陪伴感
· "免费百万安全险"——数字锚定+具体措施，信息密度高，用事实建立信任

**变体 B：条件保障型**（强调安全机制）

示例：不解锁付不了，每笔支付都会确认 → 金融级安全防控 + 全额赔付 → [了解保障]

语言特征：FORMAL=4, WARMTH=0
· "不解锁付不了"——双重否定强调安全机制，理性陈述而非软化
· "金融级安全防控"——专业术语+行业定语，正式度高，信任靠权威感

**变体 C：信任背书型**（强调规模和时间）

示例：3亿人都在用的安全支付方式 → 不解锁付不了，你敢碰我敢赔 → [了解保障]

语言特征：FORMAL=2, WARMTH=+2
· "3亿人都在用"——社交验证数字前置，从众暗示降低信任门槛
· "你敢碰我敢赔"——口语化对仗承诺，温暖+亲密感，叠韵节奏增强记忆

### 参考案例

- 碰一下："解锁手机才能付"——❌ "才能"暗示限制，应改为"解锁手机就能付"（支付宝碰一下支付实测）
- 碰一下："别人碰不了"——❌ 模糊，同时暗示功能不可用和钱安全但两个都没说清（支付宝碰一下支付实测）
→ 更多案例见 copy-style-gallery.md

## 八、失败重试

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 标题 | 6-15 字 | reassure + 具体改善，不是"已优化" |
| 正文 | 10-25 字 | 正向预期 + 改善证据 |
| CTA | 2-5 字 | 行动引导，可用叠词软化 |

### 信息递进

```
改善事实 → 正向预期 → 行动引导
```

### 必须遵守的规则

1. **reassure 语调**：对失败用户不说"高达99.4%"（暗示还有0.6%会失败），说"成功率更高了"
2. **具体改善 > 空洞表达**："成功率提升到99.4%" > "已优化" → SKILL.md 韵律检查
3. **不用"高达"对失败用户**：惊喜语气对刚失败过的用户不真诚 → SKILL.md 韵律检查
4. **禀赋效应**："你的手机已升级" > "本机型已优化" → `grammar-templates.md`§十七
5. **CTA 可用叠词**："再碰碰看""再试试" → `function-words-table.md`§五

### 变体示例

**变体 A：改善事实型**

示例：[功能名]已升级 → 现在成功率更高了 → [再试一次]

语言特征：FORMAL=3, WARMTH=0
· "已升级"——中性陈述改善，不加软化词，正式度中等
· "再试一次"——标准CTA不含叠词，"一次"限定频率，理性克制

**变体 B：禀赋效应型**

示例：你的手机已升级 → [功能名]成功率更高，现在就能成功 → [去试试]

语言特征：FORMAL=2, WARMTH=+2
· "你的"——物主代词触发禀赋效应，归属感降低重试心理门槛
· "去试试"——叠词CTA+"试"降低承诺感，"就能"正向框架暗示低门槛

**变体 C：鼓励重试型**（温暖语调）

示例：[功能名]更稳定了 → [操作步骤] → [再碰碰看]

语言特征：FORMAL=1, WARMTH=+3
· "更稳定了"——"了"语气词软化陈述，比"已升级"更口语更温暖
· "再碰碰看"——叠词"碰碰"+"看"试探语气，最大温暖度，鼓励而非命令

### 参考案例

- 碰一下："碰一下已升级，现在碰就能成功"——✅ "就能"正向框架（支付宝碰一下支付实测）
- 碰一下："成功率高达99.4%"——❌ "高达"对失败用户不真诚（支付宝碰一下支付实测）
- 碰一下："再碰碰看"——✅ 叠词+行动感，适合重试场景（支付宝碰一下支付实测）

## 九、设置 / 开关

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 功能名称 | 2-6 字 | 名词短语，说清功能是什么 |
| 功能描述（可选） | 5-15 字 | 补充说明开启/关闭后的效果 |
| 开关标签 | — | 不额外添加文案，开关本身即交互 |

### 信息递进

```
功能名称 → [效果说明] → 开关操作
```

### 必须遵守的规则

1. **功能名称用名词，不用动词**："自动更新" > "自动更新开关"——开关本身已是操作
2. **描述说效果，不说操作**："充电时自动备份" > "开启后将在充电时自动备份"
3. **开关文案不重复功能名**：功能名是"深色模式"，开关旁不需要再写"开启深色模式"
4. **正向表述**："自动播放" > "禁止手动播放"——开关开=功能启用，不要用双重否定
5. **关闭状态的后果提示**：如果关闭有重大影响，在关闭后显示 Toast 提示——"已关闭，XX功能将不可用"

### 变体示例

**变体 A：简洁型**（功能名自解释）

示例：自动更新 → [开关]

语言特征：FORMAL=3, WARMTH=0
· 仅功能名——信息密度最低，无冗余描述，适合自解释功能
· 无叠词、无语气词、无敬语——中性极简，开关操作不需要温暖感

**变体 B：描述型**（功能名需补充说明）

示例：智能省电 → 充电时自动降低性能以延长电池寿命 → [开关]

语言特征：FORMAL=4, WARMTH=-1
· "自动降低性能以延长电池寿命"——完整技术说明，"以"连接目的，高信息密度+高正式度
· 无叠词无软化——理性陈述效果而非感受，正式度高于温暖度

**变体 C：确认型**（关闭有重大影响）

示例：消息通知 → 接收新消息时显示通知 → [开关] → 关闭后提示：关闭后将无法及时收到新消息

语言特征：FORMAL=4, WARMTH=-2
· "关闭后将无法及时收到新消息"——"将无法"条件式后果预告，正式严谨不回避负面
· 双"关闭后"重复结构——强调因果关系，完整标点，信息密度最高

### 常见错误

| 错误 | 问题 | 正确 |
|------|------|------|
| "开启深色模式" | 动词+功能名，开关本身已是操作 | "深色模式" |
| "关闭后将无法收到通知" | 默认展示关闭后果，制造焦虑 | 仅在关闭后 Toast 提示 |
| "禁止自动播放" | 负向表述，双重否定 | "自动播放"（开关关=禁止） |
| "Wi-Fi" | 技术术语，用户不熟悉 | "无线网络"（或根据用户群体选择） |

## 十、支付场景

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 金额 | — | 突出显示，数字锚定 |
| 商品/服务名 | 4-10 字 | 明确用户在为什么付钱 |
| 顾虑消除 | 5-15 字 | 消除用户最大顾虑 |
| CTA | 2-8 字 | 含金额的行动按钮 |

### 信息递进

```
金额 → 商品名 → [顾虑消除] → 含金额的CTA
```

### 支付场景三要素顾虑消除模板

支付场景用户有三大核心顾虑，必须逐一消除：

| 顾虑 | 用户心理 | 消除文案 | 适用类型 |
|------|---------|---------|---------|
| 金额安全 | "会不会多扣？" | "仅扣¥XX""不会多扣" | UI/MKT |
| 退款保障 | "能退吗？" | "7天无理由退款""随时可退" | UI/MKT |
| 信息安全 | "支付信息安全吗？" | "支付信息加密保护" | UI |

**三要素组合模板**：

**变体 A：简洁型**（低金额场景）

语言特征：FORMAL=3, WARMTH=0
· "立即支付"——"立即"行动感动词，正式度中等，不含叠词或软化
· 无顾虑消除文案——信息密度最低，信任靠金额透明度而非文字安抚

**变体 B：顾虑消除型**（中高金额场景）

语言特征：FORMAL=3, WARMTH=+1
· "7天无理由退款"——数字锚定+具体保障，"无理由"比"可退"更明确更安心
· 顾虑消除文案前置——信息密度高于简洁型，温暖感来自消除担忧而非语气

**变体 C：完整保障型**（高金额/首次支付场景）

语言特征：FORMAL=4, WARMTH=-1
· "支付信息加密保护"——技术术语"加密保护"，正式度高，信任靠权威而非温暖
· 三重保障"·"并列——分隔符营造条款感，信息密度最高，正式感>亲密感

### 必须遵守的规则

1. **CTA 必须含金额**："立即支付 ¥99" > "立即支付"——按钮上看到金额减少焦虑
2. **金额格式统一**：同一支付流程中"¥99"和"99元"不能混用
3. **退款政策前置**：有退款保障时必须在支付确认页显示，不能藏在链接里
4. **不制造紧迫感**：支付场景不用"仅剩X件""最后机会"——已进入支付流程，紧迫感反而增加犹豫
5. **错误提示具体**："余额不足" > "支付失败"——告诉用户具体原因 → `grammar-templates.md`§六.2

## 十一、成功通知

### 结构

| 元素 | 字数 | 说明 |

**变体 C：庆祝型**（品牌调性活泼的产品）

示例：搞定啦！ → [去看看]

语言特征：FORMAL=1, WARMTH=+3
· "搞定啦"——口语化+"啦"语气词，轻松庆祝感，与"已保存"的正式度形成极端对比
· "去看看"——叠词CTA，温暖引导下一步；感叹号为全场唯一，庆祝场景克制使用

## 十二、失败挽回

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 问题 | 4-10 字 | 说清出了什么问题 |
| 原因（可选） | 5-15 字 | 只在帮助理解时添加 |
| 改善说明 | 5-15 字 | 告诉用户现在有什么改善 |
| CTA | 2-5 字 | 引导重试 |

### 信息递进

```
问题 → [原因] → 改善说明 → CTA
```

**与错误提示的区别**：错误提示是首次失败，失败挽回是**重复失败后**的再次引导。关键差异是"改善说明"——用户已经失败过，需要知道"这次有什么不同"。

### 必须遵守的规则

1. **reassure 语调**：对失败用户不说"高达99.4%"（暗示还有0.6%会失败），说"成功率更高了"
2. **具体改善 > 空洞表达**："成功率提升到99.4%" > "已优化" → SKILL.md 韵律检查
3. **不用"高达"对失败用户**：惊喜语气对刚失败过的用户不真诚 → SKILL.md 韵律检查
4. **禀赋效应**："你的手机已升级" > "本机型已优化" → `grammar-templates.md`§十七
5. **CTA 可用叠词**："再碰碰看""再试试" → `function-words-table.md`§五
6. **给具体改善证据**：如果有的话——"已升级网络连接方式" > "已优化"

### 变体示例

**变体 A：改善事实型**

示例：视频生成已升级 → 现在成功率更高了 → [再试一次]

语言特征：FORMAL=3, WARMTH=0
· "已升级"——中性陈述改善事实，与"更稳定了"的口语化形成对比
· "再试一次"——标准CTA不含叠词，"一次"限定频率，理性克制

**变体 B：温暖鼓励型**（品牌调性活泼的产品）

示例：差一点点 → 可能是网络不太顺畅，再碰碰看？现在成功率更高了 → [再碰碰看]

语言特征：FORMAL=1, WARMTH=+3
· "差一点点"——叠词"一点点"软化失败打击，比"失败"温和十倍
· "再碰碰看"——叠词"碰碰"+"看"试探语气，最大温暖度；问号增加互动感而非命令感

**变体 C：严谨解法型**（金融/安全产品）

示例：交易未完成 → 银行卡余额不足，请更换支付方式或稍后重试 → [更换支付方式]

语言特征：FORMAL=5, WARMTH=-2
· "请更换支付方式或稍后重试"——"请"敬语+完整并列分句，正式度最高
· "交易未完成"——"未完成"而非"失败"，专业中立术语，完整标点不省略

## 十三、系统通知（异地登录/安全提醒/信息变更）

### 结构

| 元素 | 字数 | 说明 |
|------|------|------|
| 事件 | 4-10 字 | 说明发生了什么 |
| 详情 | 10-25 字 | 关键信息（时间/地点/设备/IP） |
| CTA | 2-5 字 | 确认/修改/查看 |

### 信息递进

```
事件 → 详情（时间+地点+设备）→ CTA
```

### 必须遵守的规则

1. **中性陈述，不制造恐慌**："检测到新设备登录" > "您的账号可能被盗！"
2. **给具体信息**：时间、地点、设备类型、IP——用户凭此判断是否本人操作
3. **给两条路径**：是本人→确认；非本人→修改密码/冻结账号
4. **不用感叹号**：安全通知的感叹号=不专业=不可信 → SKILL.md 标点规则
5. **不用"请立即"**：用"如非本人操作请修改密码" > "请立即修改密码！"——后者制造恐慌
6. **"我们"可用于安全通知**：系统故障/安全事件需要责任归属 → `grammar-templates.md`§二

### 变体示例

**变体 A：信息告知型**（非紧急）

示例：新设备登录提醒 → 你的账号于今天14:30在北京登录 → [查看详情]

语言特征：FORMAL=3, WARMTH=0
· "于今天14:30在北京登录"——"于"介词+时间地点完整交代，信息密度高但不制造紧迫感
· "查看详情"——中性行动引导，不含"立即"等紧迫词，区别于紧急场景

**变体 B：行动引导型**（疑似异常）

示例：异地登录提醒 → 你的账号在异地登录，如非本人操作请修改密码 → [是我本人] / [修改密码]

语言特征：FORMAL=4, WARMTH=-1
· "如非本人操作请修改密码"——"请"敬语+条件句式，正式度高，比变体A多一层行动引导
· 双按钮"是我本人/修改密码"——路径完整度最高，"请"增加礼貌度但不亲密

**变体 C：紧急处理型**（确认异常）

示例：账号安全提醒 → 检测到你的账号存在异常操作，已临时限制部分功能 → [立即验证]

语言特征：FORMAL=4, WARMTH=-2
· "已临时限制部分功能"——"已...限制"客观陈述已采取措施，不制造恐慌但信息严肃
· "立即验证"——安全场景中"立即"合理使用，区别于一般场景的"请立即修改"制造恐慌

| 严谨合规 | "异地登录提醒" | "检测到您的账号于2026年5月21日在异地登录，如非本人操作请立即修改密码" | 立即修改 |
